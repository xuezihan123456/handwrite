# HandWrite — AI 中文手写体生成器设计文档

> 日期: 2026-04-06
> 状态: 已确认

## 一句话定位

输入中文文字，输出逼真的手写风格图片/文档。内置多种手写风格，支持整页生成、单字生成、字体文件导出。基于条件 GAN（zi2zi 架构），本地训练和推理。

## 用户画像

- Python 开发者，熟悉 PyTorch/CV
- 做科研/竞赛，需要手写风格文档
- RTX 4060 Laptop（8GB VRAM），Windows，CUDA 12.4
- 已安装：PyTorch 2.6、torchvision、OpenCV、Pillow

## 核心功能

1. **单字生成**：输入一个汉字 → 输出 256×256 手写风格图片
2. **整页生成**：输入一段文字 → 输出 A4 页面的手写文档图片（自然排版）
3. **多种内置风格**：5 种预训练手写风格可选
4. **Python 库 + Web Demo**：`import handwrite` 代码调用 + Gradio 网页界面

## 推迟的功能（不在当前设计范围）

| 功能 | 推迟到 | 原因 |
|------|--------|------|
| 自定义风格（用户上传样本训练） | v0.3 | few-shot 字体生成是独立的研究问题，50张样本简单fine-tune会过拟合 |
| TTF 字体文件导出 | v0.5 | 需要全量字符覆盖(6763字)+矢量化质量控制，工程量大 |
| 图片文字替换（OCR+替换） | v0.5+ | 技术复杂度最高(OCR+inpainting+风格匹配)，依赖核心模块成熟 |
| PDF 导出 | v0.2 | 基于 PNG 套 reportlab，不难但非 MVP 必须 |

## Style Engine — 核心 AI 模块

### 模型架构：基于 zi2zi 的条件 GAN

**Generator（生成器）**：
- U-Net 结构 + skip connections
- 8 层 encoder：64→128→256→512→512→512→512→512
- 8 层 decoder：对称结构，带 skip connections
- 输入：标准字体图片 256×256 灰度
- 输出：手写风格图片 256×256 灰度
- 风格控制：每种书写者风格对应一个 learnable embedding 向量，注入 bottleneck 层
- 参数量：~50M

**Discriminator（判别器）**：
- PatchGAN 架构
- 输出 30×30 的真假判定矩阵 + 风格分类
- 参数量：~5M

### 训练数据

**数据来源**：CASIA-HWDB 脱机手写数据集
- 中科院自动化所发布的公开数据集
- 约 300 个书写者，每人书写 3000+ 汉字
- 数据格式：`.gnt` 二进制文件

**数据预处理 pipeline**：
```
CASIA-HWDB .gnt 文件
    │
    ▼ 解析 .gnt 为 (Unicode标签, 灰度位图) 对
    │
    ▼ 过滤质量差的样本（太小、太模糊、标注错误）
    │
    ▼ 统一缩放 + 居中到 256×256，保持长宽比，白色填充
    │
    ▼ 用 Pillow + 宋体字体渲染对应的标准字体图片 256×256
    │
    ▼ 输出三元组: (标准字体图, 手写图, 书写者ID)
```

**数据量预估**：约 100-300 万配对样本

**配对数据构建注意事项**：
- 字符居中对齐：手写图和标准图必须采用相同的居中和 padding 策略
- 留白比例统一：字符占图片面积约 70-80%，周围均匀留白
- 质量过滤：去除笔画不完整、严重变形的样本
- 书写者筛选：优先选取字迹稳定、覆盖字数多的书写者

### 训练配置

| 参数 | 值 |
|------|-----|
| 分辨率 | 256×256 灰度 |
| Batch size | 8-16（视显存占用调整） |
| 优化器 | Adam, lr=2e-4, betas=(0.5, 0.999) |
| G 损失 | 对抗损失 + L1 像素损失(λ=100) + 风格分类损失 |
| D 损失 | 对抗损失 + 风格分类损失 |
| 训练技巧 | Label Shuffling（zi2zi 提出，防止 D 过强） |
| 训练轮数 | 20-30 epochs |
| 预估显存 | ~5-6 GB（RTX 4060 8GB 充足） |
| 预估训练时长 | 1-3 天 |

### MVP 字符范围

- **v0.1**：500 个最常用汉字 + 常用标点符号 + 数字 + 英文字母
- 未覆盖的字 fallback 到开源手写字体
- v0.2 扩展到 1000 字，v0.4 扩展到 3500 常用字

### 内置风格

从 CASIA-HWDB 中精选 5 种差异大的书写者：

| 风格名称 | 特征 |
|----------|------|
| 工整楷书 | 端正规范，笔画清晰 |
| 圆润可爱 | 圆润饱满，偏可爱 |
| 行书流畅 | 略带连笔，书写流畅 |
| 偏瘦紧凑 | 字形偏窄，笔画紧凑 |
| 随意潦草 | 放松随意，略带潦草 |

## Composer — 排版模块

将 Style Engine 生成的单字图片排版成自然的手写页面。

### 核心参数（带随机扰动模拟真实手写）

| 参数 | 基础值 | 随机范围 | 说明 |
|------|--------|----------|------|
| 字间距 | 字宽×0.1 | ±15-25% | 字与字之间的水平间距 |
| 行间距 | 字高×0.5 | ±5-10% | 行与行之间的垂直间距 |
| 单字大小 | 基础字号 | ±3-8% | 每个字的缩放比例 |
| 单字旋转 | 0° | ±1.5-3° | 每个字的微小旋转 |
| 基线偏移 | 0px | ±2-5px | 每个字在基线上下的偏移 |
| 行倾斜 | 0° | ±0.3-0.8° | 整行的轻微上下倾斜 |
| 首行缩进 | 2字宽 | ±3-5px | 段首空两格 |

### 墨水浓淡模拟

- 对单字图像做 alpha 通道调节
- 每 10-20 个字为一个周期，浓度在 0.7-1.0 之间正弦波动
- 模拟蘸墨/换笔的自然变化

### 纸张模板

内置 4 种纸张背景：
1. 白纸（干净无格线）
2. 横线纸（如作文纸）
3. 方格纸（如田字格）
4. 米字格

纸张纹理用 Pillow 绘制格线 + 轻微高斯噪声（sigma=2-5）模拟纸张纤维质感。
手写文字用 alpha blending 叠加到纸张上。

### 排版预设

| 预设 | 字间距扰动 | 旋转扰动 | 大小扰动 | 场景 |
|------|-----------|---------|---------|------|
| 工整 | ±5% | ±0.5° | ±2% | 正式文档 |
| 自然 | ±15% | ±2° | ±5% | 日常笔记 |
| 潦草 | ±25% | ±3° | ±8% | 快速记录 |

## Python Library API

```python
import handwrite

# 单字生成
img = handwrite.char("你", style="工整楷书")
img.save("output.png")

# 整段文字生成
page = handwrite.generate(
    "今天天气真不错，适合出去走走。",
    style="自然行书",
    paper="横线纸",
    layout="自然",     # 排版预设：工整/自然/潦草
    font_size=40,      # 字号（像素）
    lang="zh",
)
page.save("output.png")

# 多页长文
pages = handwrite.generate_pages(
    long_text,
    style="工整楷书",
    paper="方格纸",
)
for i, page in enumerate(pages):
    page.save(f"page_{i}.png")

# 查看可用风格
print(handwrite.list_styles())
# ['工整楷书', '圆润可爱', '行书流畅', '偏瘦紧凑', '随意潦草']
```

## 完整模型训练计划

### Phase 0：数据获取（1天）

**CASIA-HWDB 下载**：
1. 访问 NLPR 官网申请下载 CASIA-HWDB 1.0-1.2（脱机手写单字）
2. 下载文件：`HWDB1.0trn_gnt.zip`、`HWDB1.0tst_gnt.zip`（1.1/1.2 同理）
3. 总大小约 2-4 GB
4. 备选：GitHub 上搜索 "casia hwdb download"，部分镜像可直接获取

**标准字体准备**：
- 下载免费可商用的宋体字体文件（如思源宋体 NotoSerifSC）
- 用于渲染标准字体输入图片

### Phase 1：数据预处理（2-3天）

**Step 1：解析 .gnt 文件**
```
.gnt 二进制格式：
  [样本长度(4B)] [Unicode标签(2B)] [宽(2B)] [高(2B)] [位图数据(W×H B)]
  重复...
```
- 找 GitHub 上已有的 .gnt 解析器（如 zhm-real/casia-hwdb），验证能跑通
- 输出：每个字符保存为单独的 PNG 文件，按书写者ID分目录

**Step 2：质量过滤**
- 过滤太小（<32×32）或太大（>300×300）的异常样本
- 过滤几乎全白/全黑的损坏样本
- 统计每个书写者的字符覆盖率，保留覆盖 500 常用字 ≥ 90% 的书写者

**Step 3：图片标准化**
- 二值化（Otsu 阈值法）
- 计算字符 bounding box，裁剪掉多余白边
- 保持长宽比缩放到 220×220 以内
- 居中放置到 256×256 画布上，白色填充

**Step 4：渲染标准字体配对图**
- 对每个手写字符的 Unicode 标签，用 Pillow + 宋体渲染一张 256×256 的标准字体图
- 字体大小调整到与手写样本的字符面积大致匹配
- 同样居中放置

**Step 5：构建训练数据集**
```
输出目录结构：
paired_data/
├── writer_001/
│   ├── 你_standard.png    # 宋体 "你"
│   ├── 你_handwrite.png   # 书写者001手写 "你"
│   ├── 好_standard.png
│   ├── 好_handwrite.png
│   └── ...
├── writer_002/
│   └── ...
└── metadata.json          # {writer_id, char, standard_path, handwrite_path}
```

**预处理验证检查点**：
- 随机抽取 100 对配对数据目视检查：标准字和手写字是否对应正确、居中是否一致
- 统计每个书写者的样本数量，确保数据平衡

### Phase 2：风格筛选（0.5天）

从预处理后的数据中，人工挑选 5 个书写者作为内置风格：

**筛选标准**：
1. 字符覆盖率高（≥95% 的 500 常用字都有）
2. 字迹稳定（同一个字重复写的话风格一致）
3. 5 个书写者之间风格差异大（工整/圆润/流畅/紧凑/潦草）

**方法**：
- 对每个书写者，随机展示 20 个字符的手写图
- 目视评估风格特征，标记为上述 5 类
- 每类选 1 个最具代表性的书写者

### Phase 3：模型搭建（1-2天）

**基于 zi2zi-pytorch 搭建，而非从零实现**：

1. GitHub 搜索 "zi2zi pytorch"，选择 star 最多、代码质量好的复现版本
2. 验证能在 RTX 4060 上跑通原版训练（用少量数据）
3. 根据我们的需求做以下修改：
   - 输入分辨率确认为 256×256
   - 风格 embedding 数量改为 5（对应 5 个书写者）
   - 字符类别数改为 500（MVP 常用字）
   - 确认显存占用 ≤ 6GB

**如果找不到好的 zi2zi-pytorch 复现**：
- 用 pix2pix-pytorch（官方实现）作为基础
- 添加 category embedding（风格条件）和 constant loss（zi2zi 的贡献）
- 参考 zi2zi 原论文的损失函数设计

### Phase 4：训练（1-3天）

**Stage 1：预热训练（不使用 Label Shuffling）**

| 参数 | 值 |
|------|-----|
| 数据 | 5 个书写者 × 500 字 × 256×256 |
| Batch size | 8 |
| Epochs | 10 |
| G 学习率 | 2e-4 |
| D 学习率 | 2e-4 |
| G 损失 | L_adv + 100×L_L1 + L_category |
| D 损失 | L_adv + L_category |
| 预估时间 | 4-8 小时 |

**Stage 1 检查点**：
- 每 2 个 epoch 保存模型权重
- 生成 50 个样本目视检查：字形是否可辨认、风格是否有差异
- 监控 G_loss 和 D_loss 曲线：
  - D_loss 降到 ~0 → 说明 D 太强，需要降低 D 学习率或增加 G 训练次数
  - G_loss 不下降 → 检查数据加载、梯度是否正常

**Stage 2：Label Shuffling 训练**

当 Stage 1 的 D_loss 接近 0 且 G 质量停滞时，启用 Label Shuffling：
- 在每个 minibatch 中，对同一组标准字体输入，同时生成正确风格和随机打乱风格的输出
- 给 D 提供新的挑战，防止 D 过强导致 G 停止进步

| 参数 | 值 |
|------|-----|
| 在 Stage 1 基础上继续 | 加载最佳 checkpoint |
| Epochs | 10-20（追加） |
| Label Shuffling | 每个 batch 50% 正确标签 + 50% 打乱标签 |
| 预估时间 | 8-16 小时 |

**Stage 2 检查点**：
- 每 5 个 epoch 生成 100 个样本（5风格 × 20字），保存为对比图
- 人工评估：(1) 字形完整性 (2) 风格区分度 (3) 细节清晰度
- 如果出现模式崩塌（输出全是噪声/同一个字），回退到上一个 checkpoint 并降低学习率

### Phase 5：评估与调优（1-2天）

**定量评估指标**：

| 指标 | 计算方法 | 目标值 |
|------|----------|--------|
| L1 像素误差 | 生成图与真实手写图的 L1 距离 | 越低越好，作为趋势参考 |
| FID (Fréchet Inception Distance) | 生成图与真实图的分布距离 | < 50 为可接受 |
| 风格分类准确率 | 用训练好的 D 的分类头测试 | > 80% |
| 可辨认率 | 人工评估 100 个生成字，能否正确辨认 | > 90% |

**定性评估**：
- 对每种风格生成 "永和九年岁在癸丑"（包含多种笔画结构）
- 对比不同风格之间的差异是否明显
- 检查复杂字（如"藏""赢""鑫"）是否字形完整

**调优策略**：
- 如果整体模糊 → 增大 L1 损失权重、增加训练轮数
- 如果风格不区分 → 增大风格分类损失权重
- 如果笔画断裂 → 尝试加入边缘损失（Sobel 边缘检测 L1）
- 如果部分字崩坏 → 检查这些字的训练数据质量，考虑过滤或增加样本

### Phase 6：模型导出与集成（0.5天）

1. 保存最佳 Generator 权重（只需要 Generator，Discriminator 推理时不需要）
2. 导出为 TorchScript 格式（`torch.jit.script`），方便部署
3. 测试推理速度：单字生成应 < 50ms（RTX 4060）
4. 集成到 `handwrite/engine/model.py`，封装为简单接口：
   ```python
   engine = StyleEngine(weights_path="weights/generator.pt")
   img = engine.generate_char("你", style_id=0)  # 返回 PIL.Image
   ```

### 训练失败应急方案

| 问题 | 症状 | 应对 |
|------|------|------|
| 模式崩塌 | 所有输出看起来一样 | 降低 lr 至 1e-4，增加 L1 权重至 200 |
| D 过强 | D_loss=0 且 G 不改善 | 每训练 G 2次才训练 D 1次 |
| 训练不收敛 | Loss 震荡不下降 | 检查数据加载是否正确、梯度裁剪至 1.0 |
| 显存溢出 | CUDA OOM | batch size 降至 4，启用混合精度训练(AMP) |
| 训练太慢 | 一个 epoch > 2小时 | 减少训练字符数（先用 200 字验证）、启用 AMP |

## 风险与应对

| 风险 | 严重度 | 应对 |
|------|--------|------|
| GAN 训练不稳定（模式崩塌） | 高 | 使用 zi2zi 的 Label Shuffling 技巧；保留训练 checkpoint，随时回退 |
| CASIA-HWDB 数据预处理耗时 | 中 | 先找 GitHub 上现有的 .gnt 解析器，确认能跑通再开始；预处理代码预留 3 天 |
| 低频字生成质量差 | 中 | MVP 只覆盖 500 字，未覆盖的 fallback 到开源手写字体 |
| 8GB VRAM 不够 | 低 | 256×256 + batch=8 约 5-6GB，充足；必要时降到 batch=4 |

## 竞品差异化

| 工具 | 类型 | HandWrite 的差异 |
|------|------|-----------------|
| zi2zi | 学术项目（TF 1.x） | PyTorch 现代实现 + 产品化（API+Web+排版） |
| 手写模拟器(JS) | 网页工具 | 基于深度学习，质量远超字体替换 |
| Mathpix | 商业服务 | 免费开源，本地运行，隐私安全 |
| 各种手写字体 | 静态字体文件 | AI 生成，每个字都不同，更像真人 |

核心差异：**不是字体替换（每个字长得一样），而是 AI 生成（每个字都有自然的随机变化）**。
