# Findings & Decisions

## Product Direction
- HandWrite is being pushed toward a user-facing classroom-note generator, not primarily a training workbench or SDK.
- The first wedge is continuous paragraph classroom notes.
- The default style target is natural and flowing, readable, slightly imperfect, and more real than neat.
- The first realism problem to attack is print-looking single-character output.

## Execution Decisions
- This iteration will ship the product system, not the full 2000+ built-in prototype body.
- The implementation shape is hybrid: prototype-library foundation first, minimal precheck/report loop second.
- The system must stay honest: no silent degradation, no auto-rewriting of user text.
- Multi-agent parallel execution is explicitly authorized by the user, with GPT-5.4 + xhigh requested for worker lanes.

## Architecture Findings
- `StyleEngine` currently routes supported chars through a model path and otherwise falls back to reference rendering or a placeholder path.
- `demo/app.py` already has a stable multi-page preview/download loop, so the safest product upgrade is to add a precheck/report panel instead of rebuilding the export flow.
- `pyproject.toml` currently lacks package-data configuration, so starter prototype assets will require packaging updates.
- The environment actually contains handwriting-ish system fonts: `simkai.ttf`, `STKAITI.TTF`, and `STXINGKA.TTF` under `C:/Windows/Fonts`.
- These fonts can improve the fallback path, but runtime logic must remain resilient when they are missing.

## Constraints
- Do not commit large datasets, weights, or machine-specific paths.
- The repo must remain runnable without real generator weights.
- Current Windows environment has temp-directory permission quirks, so pytest should keep using controlled writable temp roots when needed.


## Next Product Gap (2026-04-12)
- The starter pack and local builder already existed, but the public API and demo were not yet consuming user-built packs.
- The most valuable next slice is an end-to-end custom pack loop: `build_prototype_library.py` -> `inspect_text(...)` / `generate(...)` / demo.
- That requires engine caching and inspection reporting to stay aware of which prototype pack is active.


## Custom Prototype Pack Product Loop (2026-04-12)
- `prototype_pack` is now the canonical public/demo parameter. It accepts either a pack directory or a direct `manifest.json` path.
- The most important product loop is now concrete: build local pack -> point API/demo at `prototype_pack` -> inspect coverage -> generate/export.
- Runtime reports now include `prototype_pack_name`, `prototype_source`, and `prototype_source_kind`, so docs and demo can explain which source is active.
- Engine cache had to become config-aware so built-in starter pack and custom packs do not share the same singleton runtime instance.
